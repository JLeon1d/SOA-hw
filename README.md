# Дз 1 по СОА

## Архитектура

Все юзеры и продавцы имеют уникальные внутренние (не доступны извне) идентификаторы (uuid), также все товары и заказы тоже имеют уникальные идентификаторы. В бд одного сервиса могут хранится uuid сущностей другого сервиса (НЕ foreign key между бд, но по сути тоже самое).

Сервисы как и все бд отвечают асинхронно на read запросы и синхронно на модифицирующие запросы (то есть просто не разрешается менять состояние одновременно).

### API

Пользователи отправляют запросы по https на Router, он аунтефицирует пользователей (если необходимо для запроса) и пересылвает запрос в нужный сервис. Сами сервисы взаимодействуют по gRPC.

### Profiles (/profile)

Пользователи и продавцы имеют профиля и все их перснональные данные хранятся тут. Для пользователей ещё хранится корзина (набор uuid товаров, превьюшки, названия). Для продавцов хранятся все товары которые они продают.

### Товары (/item)

Ну просто все товары (цена, фотки, отзывы). Также какие-то дополнительные данные для рекомндаций, чтобы получить ленту рекомендаций нужно прийти сюда с uuid пользователя (все статистики и предпочтения пользователей хранятся тут).


### Транзакции (/transaction)

Для всех покупок и пополнений (покупки приходят через api, пополнения прилетают от банков и т.п). Хранятся просто транзакции, то есть баланс пользователя определяется как сумма по всем его транзакциям.

### Заказы (/orders)

Всё про заказ, ходит в сервис нотификаций при обновлении статуса заказа.

### Notifications

Внутренний сервис для уведомлений (когда статус заказа обновляется или когда пользователь ошибся и не выключил рекламу).

## Обоснования архитектуры

0. Если делать не монолит, то транзакции очевидно отдельный сервиc, транзакция зависит от юзера и от заказа (в транзакцию можно не класть uuid товара, а только в историю заказов положить uuid транзакции).

1. Сервис уведомлений по понятным причинам отдельный сервис - туда нужно не часто ходить (на обновление статуса каждого заказа), в остальном он живет свой жизнь в бэкграунде.

2. Заказы впринципе можно объеденить с транзакциями (но бд-шки всё равно разные должны быть), так как каждый заказ пораждает транзакцию, но кажется удобнее транзакции всё таки сделать отдельным сервисом - транзакции хочется видеть безопасными и независимыми, не хочется иметь лишний код объединяющий их с чем-то. К тому же заказ это всё равно сложный запрос и доп. поход в транзкации не критичен (паралельно с ним можно пойти взять данные товара, обновить статистику пользователя).

3. Юзеры и товары логично разделить (разные сущности), единственное что не очевидно - где хранить вещи необходимые для ленты (aka фида) товаров (это всякие статистики по пользователю). Кажется, что лучше их хранить рядом со всеми товарами, так удобнее сразу выдавать ленту товаров без дополнительной пересылки данных (причем больших данных).
